# システム管理画面からの「完全削除（物理削除）」機能の実装計画

システム管理画面から、町内会および関連する全てのデータをデータベースから完全に消去する機能を実装します。

## 目的
* テストで作成した不要な町内会や、誤って作成された「幽霊町内会（管理者不在など）」を整理可能にする。
* 「退会（ステータス変更）」ではなく「物理削除」を行うことで、同じ町内会名・郵便番号での再登録を可能にする。
* 操作ミスによる事故を防ぐため、この機能は「システム管理者」専用とし、実行時に厳重な確認（町内会名の入力など）を求めるようにする。

## Proposed Changes

### `src/components/SystemAdminView.tsx`
システム管理画面に以下の機能を追加します。

#### [MODIFY] `src/components/SystemAdminView.tsx`
1. **「完全削除」ボタンの追加**
   * 各町内会の右端（操作列）に、赤色のゴミ箱アイコンが付いた「完全削除」ボタンを追加します。
2. **削除確認モーダル（またはプロンプト）の実装**
   * 間違ってクリックした際の事故を防ぐため、ボタンを押すと「本当に削除しますか？確認のため町内会名を入力してください」という入力付きのアラートを表示します。
3. **完全削除ロジック（Cascading Delete）の実装**
   * データベースの外部キー設定（CASCADE）が不完全な場合を考慮し、システムが安全に子データから順番に削除する処理を実装します。
   * 削除の順番（予定）：
     1. その町内会に属するお知らせ（`circulars`）に紐づく出欠データ（`event_applications`）と既読データ（`read_receipts`）の削除
     2. お知らせデータ（`circulars`）の削除
     3. 名簿データ（`resident_rosters`）の削除
     4. 役員データ（`neighborhood_admins`）の削除
     5. システム設定（`system_settings`）および請求データ（`platform_payments`）の削除
     6. 最後に、町内会本体（`neighborhoods`）の削除
4. **画面の更新**
   * 削除完了後、一覧データを再取得して表示を更新します。

## User Review Required

> [!WARNING]
> この処理を実行すると、対象の町内会データは完全に消滅し、復旧できなくなります。
> システム管理者専用の機能とはいえ、強力な権限となります。削除実行時には「町内会名のタイピングによる最終確認」を挟む予定ですが、それで問題ないでしょうか？

問題なければ、「承認」をお知らせください。実装に取り掛かります。
